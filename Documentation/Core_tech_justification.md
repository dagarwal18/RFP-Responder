# Core Technology Justification: RFP-Responder

## 1. Orchestration Layer
- **Tech Used:** LangGraph (14-Node State Machine)
- **Alternatives Available:** Temporal.io, Apache Airflow, AutoGen, CrewAI.
- **Justification:** Enterprise orchestration tools like Temporal or Airflow are powerful but overly complex for managing purely LLM state graphs and lack native AI framework integration. Frameworks like AutoGen and CrewAI abstract too much control away. LangGraph strikes the exact balance: it provides explicit directed-graph state management while retaining granular, Python-level control over conditional edges, enabling deterministic retry loops and hard "Legal Veto" routing paths.

## 2. Fast Structural Extraction (LLM Engine)
- **Tech Used:** Qwen3-32B via Groq Cloud (LPU Inference)
- **Alternatives Available:** Claude 3.5 Haiku, GPT-4o-mini, Gemini 1.5 Flash.
- **Justification:** While Haiku and GPT-4o-mini offer exceptional structural reasoning, Groq's dedicated LPU hardware guarantees deterministic millisecond latency—critical when running 10+ rapid sub-graph tasks per RFP. Furthermore, utilizing an open-weights model (Qwen3) ensures long-term data privacy compliance for extracting sensitive corporate pricing data, avoiding strict vendor lock-in with closed-source APIs.

## 3. Deep Narrative & Cross-Validation (LLM Engine)
- **Tech Used:** Llama 4 Scout 17B
- **Alternatives Available:** Claude 3.5 Sonnet (200k), GPT-4o (128k), Gemini 1.5 Pro (1M+).
- **Justification:** Proprietary frontier models provide massive context windows, but recursivly feeding 50-page RFPs into commercial APIs through multi-agent validation loops is financially prohibitive. Llama 4 Scout (17B) provides a highly performant ~131K context window that can be run cost-effectively, strictly enabling aggressive cross-document semantic validation without the cost penalties of proprietary APIs.

## 4. Multimodal Table Parsing
- **Tech Used:** HuggingFace Qwen3-VL-8B-Instruct
- **Alternatives Available:** AWS Textract, Azure Document Intelligence, Google Document AI, Unstructured.io.
- **Justification:** Enterprise OCR services (Textract, Azure) perform well on standard tax forms but struggle contextually with complex, multi-page B2B SLA matrices containing merged cells and domain-specific jargon. Qwen3-VL natively understands the *relationship* between the structural layout and the linguistic context, extracting perfectly formatted JSON payloads rather than raw, disconnected text blocks.

## 5. Vector Search Engine
- **Tech Used:** Pinecone Serverless + BAAI/bge-m3 Embeddings
- **Alternatives Available:** pgvector (PostgreSQL), Qdrant, Milvus, Weaviate.
- **Justification:** `pgvector` provides a unified SQL stack, but Pinecone Serverless entirely removes the need to manage infrastructure scaling for spiky, high-load RFP ingestion workloads. BAAI/bge-m3 was explicitly selected over proprietary embeddings (e.g., OpenAI `text-embedding-3`) because it demonstrates superior zero-shot retrieval on dense, multi-lingual technical and engineering text.

## 6. Exact Keyword & Relational Storage
- **Tech Used:** MongoDB + BM25 Search
- **Alternatives Available:** PostgreSQL (JSONB + pg_trgm), Elasticsearch, Redis.
- **Justification:** Elasticsearch offers the best raw BM25 performance but requires complex JVM infrastructure. While PostgreSQL is robust, MongoDB's flexible document schema maps seamlessly to the highly variable, unpredictable JSON structures generated by LLMs during raw RFP extraction. Its native Atlas search capabilities easily handle the deterministic keyword retrieval required mathematically block hallucinated rate cards.

## 7. Backend API & Telemetry
- **Tech Used:** FastAPI, Uvicorn, WebSockets, Pydantic v2
- **Alternatives Available:** Node.js (Express), Go (Fiber), Python gRPC.
- **Justification:** A Node.js or Go backend would deliver superior concurrent throughput, but because the entire AI orchestration ecosystem (LangGraph, Transformers) is Python-centric, bridging languages introduces severe latency and serialization overhead. FastAPI provides class-leading async I/O *for Python*, native Pydantic typing for strict agent state validation, and scalable WebSockets to stream step-by-step telemetry to the frontend user.

## 8. Frontend Review Interface
- **Tech Used:** Next.js 16, React 19, TailwindCSS v4
- **Alternatives Available:** Vue 3 (Nuxt), SvelteKit, HTMX + Alpine.js.
- **Justification:** SvelteKit or HTMX would yield smaller bundle sizes, but constructing the heavily stateful, complex split-pane diff editors required for the "Human Validation Gate" heavily benefits from React's mature component ecosystem (e.g., Shadcn). React 19’s concurrent rendering explicitly solves UI thread-locking issues when diffing and rendering massive 10,000+ word AI-generated proposal arrays in real time.

## 9. Submission Formatting
- **Tech Used:** Custom Python (`md_to_pdf.py`) + Mermaid-CLI (`npx`)
- **Alternatives Available:** Pandoc, headless Chrome (Puppeteer), LaTeX, ReportLab.
- **Justification:** Pandoc is the industry standard but fails to gracefully handle malformed markdown syntax dynamically generated by LLMs (e.g., unclosed tables). LaTeX is too rigid for unpredictable AI generation. Our custom Python adapter acts as an active sanitization layer—performing table row deduplication and syntax stripping—before passing it via Mermaid-CLI to securely render technical architecture diagrams directly into the final PDF.
