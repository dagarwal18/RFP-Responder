## Instructions

1. Read the RFP structure above carefully. Identify EVERY section that the response document must contain — including sections not driven by specific requirements (e.g., cover letter, company profile, case studies, client references, submission forms, certifications).

2. For each section, assign a `section_type`:
   - `"requirement_driven"` — sections that directly address extracted requirements (e.g., Technical Solution, SLA Commitments, Compliance Matrix)
   - `"knowledge_driven"` — sections populated from company knowledge base, not tied to specific requirement IDs (e.g., Company Profile, Case Studies, Client References, Team Qualifications)
   - `"commercial"` — pricing and commercial terms sections (handled by the Commercial Agent)
   - `"legal"` — legal terms, deviations, contract clauses (handled by the Legal Agent)
   - `"boilerplate"` — cover letter, table of contents, submission forms, declaration pages

3. For `requirement_driven` sections:
   - Assign REPRESENTATIVE requirement IDs (3-5 per section) that best exemplify
     what the section covers. You do NOT need to list every single requirement ID —
     a separate gap-filling process will complete the mapping after you.
   - Focus on correctly categorizing WHICH TYPES of requirements belong in each section.
   - Map relevant company capabilities to each section.
   - SLA / PERFORMANCE METRICS: Requirements about availability, uptime, latency,
     jitter, packet loss, call setup time, response time, MTTR, RPO, RTO, or any
     measurable SLA target → assign to an SLA / Non-Functional Compliance section.
   - VENDOR QUALIFICATIONS: Requirements about certifications, years of experience,
     team qualifications, references, or vendor eligibility → assign to
     Company Profile, Certifications, or Team Qualifications sections.
   - EVALUATION CRITERIA: Requirements about scoring, evaluation, demonstrations,
     or selection criteria → assign to a Compliance Matrix or Evaluation section.

4. For `knowledge_driven` sections:
   - Map relevant company capabilities / evidence that should be cited
   - Provide clear description of what content should be included

5. For `commercial`, `legal`, and `boilerplate` sections:
   - Leave `requirement_ids` empty
   - Leave `mapped_capabilities` empty (these will be handled by their dedicated agents)
   - Provide a clear description of what the section should contain

6. Assign a priority (1 = first section in the document) to each section based on the DOCUMENT ORDER that a reader should follow. This must reflect the logical reading order of a professional proposal: Cover Letter (1), Executive Summary (2), Table of Contents (3), then the Technical Implementation section and its subsections, then Company Profile, then Commercial, then Legal, then Appendices (last). Priority numbers MUST be sequential (1, 2, 3, ...) and MUST match the order you want the sections to appear in the final assembled document.

7. Provide a `description` for EVERY section explaining what it must contain.

8. Design an ideal, professional proposal structure that is INFORMED BY the RFP's content but is NOT a 1:1 copy of the RFP's table of contents. Analyze the RFP's topics, scope, and requirement types to decide which proposal sections are needed (e.g., Technical Architecture, Security & Data Protection, Operational Support, Compliance, etc.). The section titles and grouping should reflect what a winning proposal for THIS SPECIFIC RFP would look like — not a hardcoded generic template and not a blind copy of the RFP's own headings.

9. **Consolidate Requirement Matrices**: Do NOT create multiple separate "Requirement Matrix" or "Compliance Matrix" sections broken down by requirement class (e.g., separate sections for "Matrix — Security", "Matrix — Operations", "Matrix — Technical"). Instead, embed each group of requirements directly into the relevant logical section (e.g., security requirements go into the Security & Data Protection section as a compliance table within that section's content). Only create a standalone matrix section if the RFP explicitly demands a single unified compliance matrix appendix.

10. For EVERY section, decide whether the final proposal should include a visual. Set:
   - `visual_relevance`: `"none"`, `"optional"`, or `"required"`
   - `visual_type_hint`: `"architecture"`, `"sequence"`, `"gantt"`, `"state"`, `"journey"`, `"class"`, or `"auto"`
   - `visual_notes`: a short description of what the visual should show
   - `visual_source_terms`: 3-8 terms, entities, platforms, workstreams, or phases that should appear in the visual if one is generated
   Rules:
   - Technical architecture, deployment topology, cloud interconnect, integration, workflow, operational lifecycle, and implementation timeline sections usually deserve a visual.
   - Pricing tables, legal terms, declaration forms, appendices, cover letters, executive summaries, and table-of-contents sections should usually be `visual_relevance = "none"`.
   - Use `gantt` only for schedule, rollout, migration, or phased implementation sections.
   - Use `sequence` for integrations, interfaces, hand-offs, or data exchanges.
   - Use `state` or `journey` for operational workflows, service lifecycle, escalation, onboarding, or support processes.
   - Use `architecture` or `class` for technical solution structure, platform components, topology, or service decomposition.

## Structure Detection Rule

BEFORE generating sections, analyze the RFP structure and submission instructions above.
Design an ideal proposal structure based on the RFP's content areas. Group your sections by logical domain rather than simply replicating the RFP's own headings.

CRITICAL RULE: Do NOT emit multiple disconnected top-level technical sections. The proposal should have one parent technical section such as "Technical Implementation", with granular technical domains represented as subsections inside it (for example: SD-WAN Core Architecture, Technical Compliance Matrix, Cloud Interconnects & Data Center, Managed Security Services (SOC), and IoT & Mobility).
You MUST explicitly preserve those granular technical domains, but they should sit under the single parent technical section rather than becoming separate top-level sections.

This rule takes HIGHEST PRIORITY — proposal structure must be purposeful, granular, and driven by logical domains based on the RFP context. You must absolutely avoid blindly replicating the RFP's table of contents or dumping requirements into catch-all sections.



Return a JSON object with two keys:
- `"rfp_response_instructions"`: string — any explicit instructions found in the RFP about how to format/structure the response (empty string if none found)
- `"sections"`: array of section objects

- `"source_rfp_section"`: string (which RFP section triggered this, or empty string)
