## Instructions

1. Read the RFP structure above carefully. Identify EVERY section that the response document must contain — including sections not driven by specific requirements (e.g., cover letter, company profile, case studies, client references, submission forms, certifications).

2. For each section, assign a `section_type`:
   - `"requirement_driven"` — sections that directly address extracted requirements (e.g., Technical Solution, SLA Commitments, Compliance Matrix)
   - `"knowledge_driven"` — sections populated from company knowledge base, not tied to specific requirement IDs (e.g., Company Profile, Case Studies, Client References, Team Qualifications)
   - `"commercial"` — pricing and commercial terms sections (handled by the Commercial Agent)
   - `"legal"` — legal terms, deviations, contract clauses (handled by the Legal Agent)
   - `"boilerplate"` — cover letter, table of contents, submission forms, declaration pages

3. For `requirement_driven` sections:
   - Assign REPRESENTATIVE requirement IDs (3-5 per section) that best exemplify
     what the section covers. You do NOT need to list every single requirement ID —
     a separate gap-filling process will complete the mapping after you.
   - Focus on correctly categorizing WHICH TYPES of requirements belong in each section.
   - Map relevant company capabilities to each section.
   - SLA / PERFORMANCE METRICS: Requirements about availability, uptime, latency,
     jitter, packet loss, call setup time, response time, MTTR, RPO, RTO, or any
     measurable SLA target → assign to an SLA / Non-Functional Compliance section.
   - VENDOR QUALIFICATIONS: Requirements about certifications, years of experience,
     team qualifications, references, or vendor eligibility → assign to
     Company Profile, Certifications, or Team Qualifications sections.
   - EVALUATION CRITERIA: Requirements about scoring, evaluation, demonstrations,
     or selection criteria → assign to a Compliance Matrix or Evaluation section.

4. For `knowledge_driven` sections:
   - Map relevant company capabilities / evidence that should be cited
   - Provide clear description of what content should be included

5. For `commercial`, `legal`, and `boilerplate` sections:
   - Leave `requirement_ids` empty
   - Leave `mapped_capabilities` empty (these will be handled by their dedicated agents)
   - Provide a clear description of what the section should contain

6. Assign a priority (1 = highest importance) to each section based on BUSINESS IMPACT and RFP evaluation weighting, NOT document order. Technical solutions, compliance, and SLA sections should have the highest priority (1-3). Knowledge-driven sections (company profile, references) should have medium priority (4-6). Commercial/legal sections should have mid-high priority (3-5). Boilerplate (cover letter, TOC, forms) should have the LOWEST priority (8-12). Do NOT assign sequential priorities matching the section order.

7. Provide a `description` for EVERY section explaining what it must contain.

8. If the RFP specifies how the response should be structured, follow that structure. If not, use a logical professional proposal order.

## Output Format

Return a JSON object with two keys:
- `"rfp_response_instructions"`: string — any explicit instructions found in the RFP about how to format/structure the response (empty string if none found)
- `"sections"`: array of section objects

Each section object must have:
- `"section_id"`: string (e.g., "SEC-01")
- `"title"`: string (descriptive section title for the response document)
- `"section_type"`: string ("requirement_driven" | "knowledge_driven" | "commercial" | "legal" | "boilerplate")
- `"description"`: string (what this section should contain — be specific)
- `"content_guidance"`: string (any specific RFP instructions for this section, or empty string)
- `"requirement_ids"`: list of requirement IDs (only for requirement_driven sections, empty list otherwise)
- `"mapped_capabilities"`: list of short capability descriptions (for requirement_driven and knowledge_driven sections)
- `"priority"`: integer (1 = highest priority)
- `"source_rfp_section"`: string (which RFP section triggered this, or empty string)
